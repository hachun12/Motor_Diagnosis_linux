1. 系統概述

本系統透過 DAQ 擷取馬達的 3 相電流與 3 軸振動訊號（6 通道、5 kHz）， 以 CNN 模型即時分類故障，並依 ISO 10816 振動 RMS 分區判斷。 操作介面為瀏覽器 WebUI（登入制，區分 admin / viewer 角色）， 支援即時波形、FFT 頻譜、標註存檔、歷史記錄下載與回放（回放亦可跑模型判斷）。

項目內容
服務埠8000（HTTP，可於 docker-compose.yml 修改對外埠）
資料持久化./saved_data/（記錄 CSV）、./state/（帳號、標籤、session 金鑰）
資料源模式simulator（模擬）｜nidaq（NI 硬體，僅 x86_64）｜replay（CSV 重播）
映像大小約 1.5 GB（含 PyTorch CPU 版）

2. 平台支援矩陣

目標平台Docker 部署可用資料源備註
x86_64（Ubuntu 20.04+ 等）✅ simulator、replay、nidaq* *nidaq 建議改用「部署方式 C」直接跑主機
ARM64（Ubuntu 24.04 等）✅ simulator、replay NI-DAQmx 無 ARM64 Linux 驅動；需搭配 ARM 相容 DAQ 硬體（driver 另行開發）
重要限制：NI-DAQmx for Linux 僅支援 x86_64。 ARM64 主機無法連接 NI 硬體（與 Docker 無關，是官方驅動限制）。 ARM64 現場如需實際量測，須採用具備 aarch64 Linux 函式庫的 DAQ 硬體 （如 LabJack T7/T8、MCC DAQ HAT），並請開發人員新增對應驅動模組。

3. 部署前準備

  • 目標主機已安裝 Docker Engine 與 docker compose（docker --version 確認）。
  • 取得完整專案資料夾（含 Dockerfile、docker-compose.yml、config.yaml、 app/、best_motor_cnn_weights.pth、requirements-web.txt）。
  • 確認防火牆放行服務埠（預設 8000）。
  • 首次建置需可連外網（下載基底映像與 Python 套件）；若目標機無外網，改用 §5 的「映像搬運」方式。

4. 部署方式 A：x86_64 主機（Docker）

最單純的情境：在目標機上直接建置並啟動。

cd <專案資料夾>
ADMIN_PASSWORD=請設定管理員密碼 docker compose up -d --build
# 完成後瀏覽 http://<主機IP>:8000，帳號 admin

預設資料源為 simulator（模擬訊號），適合驗收與教育訓練。

5. 部署方式 B：ARM64 主機（Docker）

方式 B-1：ARM64 主機可連外網（最簡單）

指令與方式 A 完全相同——Dockerfile 為 multi-arch，在 ARM64 上會自動抓 aarch64 版套件：

cd <專案資料夾>
ADMIN_PASSWORD=請設定管理員密碼 docker compose up -d --build

方式 B-2：由 x86_64 開發機交叉建置後搬運（目標機無外網）

在 x86_64 開發機上（一次性環境準備 + 建置匯出）：

# 一次性：安裝 ARM64 模擬與 builder
docker run --privileged --rm tonistiigi/binfmt --install arm64
docker buildx create --name multiarch --use

# 建置 ARM64 映像並打包
cd <專案資料夾>
docker buildx build --platform linux/arm64 -t motor-diagnosis:arm64 --load .
docker save motor-diagnosis:arm64 | gzip > motor-diagnosis-arm64.tar.gz

在 ARM64 目標機上（把壓縮檔與專案資料夾一起複製過去）：

docker load < motor-diagnosis-arm64.tar.gz
docker tag motor-diagnosis:arm64 motor-diagnosis:latest
cd <專案資料夾>
ADMIN_PASSWORD=請設定管理員密碼 docker compose up -d
# 注意：不要加 --build（直接使用載入的映像）
專案資料夾在目標機上仍是必要的——docker-compose.yml、 config.yaml 與資料目錄（saved_data/、state/）都掛載自該資料夾。

6. 部署方式 C：x86_64 + NI DAQ 硬體（不經 Docker）

NI 驅動需存取主機核心模組，放進容器過於複雜，此情境建議直接在主機執行：

# 1. 先依 NI 官方步驟安裝 NI-DAQmx for Linux（確認支援矩陣涵蓋你的 OS/核心版本）
# 2. 建立 Python 環境並啟動
cd <專案資料夾>
python3 -m venv .venv && source .venv/bin/activate
pip install --extra-index-url https://download.pytorch.org/whl/cpu -r requirements-web.txt
pip install nidaqmx

# 3. 設定資料源並啟動服務
DATA_SOURCE=nidaq ADMIN_PASSWORD=請設定管理員密碼 python -m app.main

裝置名稱預設 Dev1（通道 Dev1/ai0:5），不同時修改 config.yaml 的 datasource.nidaq.channels。 如需開機自動啟動，可另配 systemd unit（請洽開發人員）。

8. 首次啟動與帳號管理

  1. 瀏覽 http://<主機IP>:8000，以 admin 與部署時設定的密碼登入。
  2. 登入後點右上「改密碼」更換為正式密碼。
  3. 點「使用者管理」建立其他帳號：
    • admin：可啟動/停止、存檔、管理標籤與使用者
    • viewer：僅能觀看波形、狀態與歷史記錄（現場看板、一般人員用）
帳號資料存於 state/users.yaml（bcrypt 雜湊，非明文）。 session 有效 12 小時，逾期需重新登入。

9. 日常維運

操作指令（於專案資料夾執行）
查看服務狀態docker compose ps
查看日誌docker logs -f motor-diagnosis
重啟 / 停止docker compose restart｜docker compose down
更新版本覆蓋新版專案檔後 docker compose up -d --build（資料目錄不受影響）
備份打包 saved_data/ 與 state/ 兩個資料夾即可完整還原資料與帳號
  • 服務設定 restart: unless-stopped，主機重開機後容器會自動啟動。
  • 記錄 CSV 每 60 秒約 46 MB，請定期檢查 saved_data/ 磁碟用量。
安全提醒：本系統為區網應用（HTTP）。如需跨網段或對外提供， 請置於反向代理（Nginx/Caddy）之後加上 TLS，勿直接暴露於公網。

10. 疑難排解

症狀處置
忘記 admin 密碼 docker compose down → 刪除 state/users.yaml → ADMIN_PASSWORD=新密碼 docker compose up -d（其他帳號需重建）
埠 8000 被占用 修改 docker-compose.yml 的 ports（例如 "9000:8000"）後重啟
按「啟動」顯示 DAQ 啟動失敗 確認 DATA_SOURCE 設定正確；nidaq 模式確認驅動已裝、裝置已接、 名稱與 config.yaml 一致（預設 Dev1）
頁面異常或改版後畫面沒更新 先 Ctrl+F5 強制重新整理；再檢查 docker logs 有無錯誤
模擬模式下 AI 一直顯示異常（紅字） 正常現象——模型以真實馬達資料訓練，模擬訊號本就不會判為正常； 接上真實 DAQ 後才具參考性
回放顯示「資料不足」 該記錄長度不足 2048 點（0.41 秒），多為舊版桌面程式存的檔；新版存檔皆為完整 60 秒
登入顯示「失敗次數過多」 防爆破保護，等 60 秒後再試

故障類別代碼

模型輸出 12 類：N（正常）、RB、RB2、RBS、RBS1、RBS2、RU、RUB、RUB2、RUBS、RUBS1、RUBS2。 各代碼的工程定義待模型訓練方提供後補充於此。
